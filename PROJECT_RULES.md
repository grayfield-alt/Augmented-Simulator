# 프로젝트 개발 지침서 (PROJECT RULES)

이 문서는 본 프로젝트 진행 시 AI 코딩 어시스턴트(Antigravity 등) 및 기여자가 항상 최우선으로 준수해야 하는 공통 가이드라인입니다.

## 0. 언어 지침 (Language Policy)
- **모든 답변과 문서는 한글로 작성한다.**
- 사용자 응대, 작업 목록(`task.md`), 구현 계획서(`implementation_plan.md`), 결과 보고서(`walkthrough.md`) 등 모든 산출물에 적용된다.
- 코드 내 주석은 복잡한 로직 설명이 필요한 경우 한글로 작성한다.

## 1. 코드 스타일 (Code Style)
- **모듈 시스템**: 반드시 ES 모듈(`import`/`export`)을 사용하며, CommonJS(`require`)는 금지합니다.
- **문법**: 가능하면 구조 분해 할당(Destructuring assignment)을 적극적으로 사용합니다.
- **타입스크립트**: TypeScript 사용 시 반드시 `strict` 모드를 활성화하고 엄격한 타입 검사를 준수합니다.

## 2. 워크플로우 (Workflow)
- **타입 검사**: 코드 변경 후에는 반드시 `npm run typecheck` 명령어를 실행하여 타입 오류가 없는지 확인합니다.
- **테스트 실행**: 전체 테스트 스위트를 실행하지 말고, 관련된 단일 테스트만 실행합니다. (예: `npm test -- path/to/test`)
- **TDD 기반**: **[중요]** 항상 기능 구현 전에 테스트(Test)를 먼저 작성해야 합니다.
- **개발 환경 격리**: 루트 `proto2.html`은 빌드 배포 산출물이므로 직접 수정을 엄격히 금지합니다. 기능 개발을 위한 HTML 템플릿은 반드시 `/prototypes` 폴더 내에서 관리해야 합니다.
- **작업 인계 (Handoff)**: 세션 종료 혹은 중요한 기능 마일스톤 완료 후 커밋하기 전, 반드시 `HANDOFF.md` 파일을 최신 상태(Current Goal, Done, Next Steps 등)로 업데이트해야 합니다.

## 3. Git 컨벤션 (Git Conventions)
- **브랜치 네이밍**: 브랜치 생성 시 `feature/[작업명]` 또는 `fix/[이슈번호]` 형식을 따릅니다.
- **커밋 메시지**: 컨벤셔널 커밋(Conventional Commits) 규약을 준수합니다. (예: `feat:`, `fix:`, `docs:`, `refactor:`)
- **강제 푸시**: `main` 브랜치에 어떠한 경우에도 강제 푸시(Force Push, `-f`)를 금지합니다.

## 4. 백엔드 및 기타 주의사항
- **인증(Auth)**: `auth` 미들웨어는 `passport`가 `req.user`를 정상적으로 설정해야만 동작합니다.
- **데이터베이스**: DB 마이그레이션 작업은 반드시 `npm run migrate` 명령어를 통해서만 실행합니다.

---

## 6. 아키텍처 및 코딩 규정 (Architecture & Coding Policy)
이 프로젝트는 **클린 아키텍처(Clean Architecture)** 기반으로 코딩하며, 다음 원칙을 엄격히 준수합니다.

- **의존성 규칙(Dependency Rule)**: 고수준 정책(비즈니스 로직)은 저수준 세부사항(UI, 데이터베이스 등)에 의존하지 않아야 합니다.
- **계층 분리**:
  - **Entities**: 핵심 비즈니스 모델 및 규칙 (`src/logic/` 내 핵심 엔티티)
  - **Use Cases**: 애플리케이션 특정 비즈니스 로직
  - **Interface Adapters**: 컨트롤러, 프레젠터, 게이트웨이
  - **Frameworks & Drivers**: UI(React/HTML), 외부 API, 라이브러리 (Vite, Tailwind 등)
- **관심사 분리(Separation of Concerns)**: 각 모듈과 클래스는 하나의 책임만 가집니다.
- **명확한 인터페이스**: 컴포넌트 간 통신은 정의된 인터페이스를 통해 이루어져야 합니다.

## 7. 보편타당한 사용성 (Universal Usability)
- **레이아웃 안정성**: 콘텐츠의 동적 변화(예: 버튼 노출, 결과창 팝업)가 발생해도 핵심 게임 영역(전투 구역 등)의 크기와 비율은 고정되어야 함.
- **비율 보존**: 해상도 가변 대응 시, 원형이나 이미지 등의 시각적 요소가 찌그러지지 않도록 고정 비율(Aspect Ratio)을 엄격히 준수함.
- **입력 피드백**: 모든 버튼 및 상호작용 요소는 모바일 터치 환경을 고려하여 최소 48dp 이상의 크기를 확보하고 적절한 시각적/청각적 피드백을 제공함.
- **지속성**: 사용자 설정이나 연구소 저장 데이터는 기기 및 브라우저 환경에 관계없이 일관되게 유지되어야 함.

## 8. 디버깅 및 수정 원칙 (Debugging & Fixing Policy) (중요)
1. **추정 기반 수정 금지**: 버그 수정은 반드시 "증거(콘솔 첫 에러/스택, 네트워크 404, 재현 절차)"로 원인을 확정한 뒤에만 진행합니다.
2. **사전 원인 리포트 승인**: 코드를 수정하기 전, 항상 "원인 리포트만" 먼저 제출하여 사용자의 승인을 받습니다. (긴급 복구 예외 시 사전 고지)
   - 리포트 형식: `✅현상(1줄)` / `🧠원인(증거 포함)` / `🔎재현(3단계)` / `🛠️수정안 2개(범위 포함)` / `➜Ask(“A 혹은 B?”)`
3. **코드 임의 변경 및 푸시 금지**: 선택/승인 전에는 절대 코드를 변경하거나 커밋/푸시하지 않습니다.
4. **최소 범위 수정**: "일단 돌아가게" 하기 위해 전역 플래그, 타임아웃, 임시 try-catch를 덕지덕지 추가하는 행위를 엄격히 금지합니다.
5. **단일 목표 수정**: 한 번의 수정에는 `원인 제거` 또는 `재발 방지 장치 추가` 중 1가지만 목표로 합니다. (동시 진행은 승인 시에만)

## 9. 성능 및 구조 보호 (Performance & Structure Protection)
6. **디버그 로직 분리**: 렌더 루프 및 입력 락과 디버그 로직을 철저히 분리하며, `START_FAIL` 등의 디버깅 검사 로직이 게임 실행 자체를 막지 못하게 합니다 (경고만 출력).
7. **루프/리스너 중복 생성 금지**: 게임 루프와 리스너는 단 1개만 유지하며, `start/init` 로직은 `try/finally` 패턴을 사용하여 unlock을 보장합니다.

## 10. 보고 형식 (Reporting Format)
8. **채팅 답변 간소화**: 답변은 기본 10줄 이내로 유지하며 세부 내용은 `HANDOFF.md` 등에 기재하고 요약만 채팅에 남깁니다.
9. **디버깅 보고 템플릿**: 디버깅 보고는 항상 3문장 이내 체제를 지킵니다. `✅Fix(예정) / 🧠Cause(증거) / ➜Ask(진행 승인 질문)`
